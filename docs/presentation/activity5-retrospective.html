도메인 · 실습 목표

삼성 헬스 CSV로 연령대별 BMI 분포(%)를 계산한다.

BMI 분류 (README)

범주조건
저체중BMI ≤ 18.5
정상18.5 < BMI < 23
과체중23 ≤ BMI < 25
비만BMI ≥ 25

누락: weight==0 / height==0 → 동일 연령대 평균 보정

3. AI(Cursor) 활용 — 도움 · 한계

도움이 된 순간

  • PCTF 프롬프트 — Activity 1 스멜 분석·표 정리
  • 도메인 규칙 정렬 — README·.cursorrules 기준 BMI=25 등 경계 검증
  • 리팩토링 설계 — SRP 클래스 분리·Facade 패턴 제안
  • TDD 보조 — fixture CSV·should_…_when_… TC 초안
  • 문서화 — docs/·Report/Prompting 워크플로
  • 회귀 확인 — mvn test 실패 시 원인 추적

한계 · 주의

  • 초기 분석이 리팩토링 전 코드 기준이면 After 문서와 혼동
  • 대규모 변경을 한 커밋에 묶기 쉬움 → 단계 분리 필요
  • AI 제안 코드는 TC·도메인으로 반드시 검증
  • 프로젝트 규칙(AGENTS.md) 없으면 스타일·20줄 규칙 이탈
  • “분석만” vs “구현” 지시를 명확히 구분해야 함

4. TC가 개선에 미친 영향 · 작성 팁

영향

  • BMI=25 버그 — 경계 TC로 수정·고정 (should_classifyAsObesity_when_bmiIs25)
  • Activity 4 — 기능별 TDD 커밋(연령대·키·정상목록·전체비율)
  • 리팩토링 안전망 — 17건 → 42건으로 회귀 범위 확대
  • 작은 fixture — src/test/resources/*.csv 19개로 시나리오 분리

TC 작성 팁 (본 프로젝트)

  • 이름: should_[결과]_when_[조건] (영문, `.cursorrules`)
  • 경계값 18.5, 23, 25 각각 한 TC
  • 보정: 동 연령대만 / 타 연령대 미적용 / 유효값 0건
  • 비율: 합 100%·빈 연령대 0%·전체 vs 연령대 대비
  • 예외: 빈 파일·컬럼 부족·숫자 파싱 실패
  • 단위(BmiClassifier) + 통합(SHealth+CSV) 병행

5. 클린코드 · 리팩토링 — 장점과 어려운 점

장점

  • 클래스 이름만으로 파이프라인 이해 가능
  • 신규 기능(키 보정)을 기존 패턴에 맞춰 추가 용이
  • 스멜 문서 → 리팩토링 → TC의 추적 가능한 개선 루프
  • AI와 협업 시 규칙 파일로 품질 기준 공유

어려운 점

  • Activity 4에서 구조+기능 동시 변경 — 커밋·리뷰 부담
  • 잔여 중복(Imputer)은 “돌아가니까” 남기기 쉬움
  • legacy API(getBmiRatio(int,int))와 enum 공존
  • 문서 3종(Before/After/순서) 동기화 필요
